6. 命名约定

6.1 选择名称

给事物起能让新读者（即使是与作者不同团队的人）也能理解其用途或意图的名称.

不要为节省水平空间而担心；让新读者能立即理解你的代码比省略字符更重要.

 

考虑名称将被使用的上下文.

即使名称在离使其可用的代码很远的地方使用，也应具有描述性.

然而，名称不应通过重复出现在直接上下文中的信息来分散读者注意力.

通常，这意味着描述性应与名称的可见范围成比例.

声明在头文件中的自由函数可能应该提及头文件所属的库，而局部变量通常不需要说明它所在的函数.

 

尽量减少使用对项目外的人可能不熟悉的缩写（尤其是首字母缩略词和首字母连写词）.

不要通过在词中删除字母来缩写.

当使用缩写时，优先将其作为一个单独的“单词”大写，例如 StartRpc() 而不是 StartRPC() .

经验法则是，如果一个缩写在 Wikipedia 上有条目，通常可以接受.

注意某些普遍已知的缩写是可以接受的，例如用于循环索引的 i 和用于模板参数的 T .

 

你最常看到的名字与大多数名字不同；少数“词汇”名字被广泛重复使用，以至于它们在任何上下文中都会有意义.这些名字通常很短甚至是缩写，其完整含义来自专门的长篇文档，而不是仅仅来自对其定义的注释或名字本身的词语. 例如， absl::Status 在开发指南中有专门的一页，记录其正确用法.你可能不会经常定义新的词汇名字，但如果确实要定义，请进行额外的设计评审，以确保所选名字在被广泛使用时仍然合适.

class MyClass {
 public:
  int CountFooErrors(const std::vector<Foo>& foos) {
    int n { 0 };  // 在有限的范围和语境下含义明确
    for (const auto& foo : foos) {
      ...
      ++n;
    }
    return n;
  }
  // 函数注释不需要解释该函数在失败时会返回非 OK，
  // 这一点由 absl::Status 返回类型已隐含，

    return absl::OkStatus();
  }

  const int kMaxAllowedConnections {...};  // 在上下文中具有明确含义

6.2 文件命名

文件名应全部小写，可包含下划线 ( _ ) 或连字符 ( - ).

遵循你项目所使用的约定.

如果没有可遵循的一致本地模式，优先使用 " _ ".

可接受的文件名示例：

xxxxxxxxxx
my_useful_class.cc
my-useful-class.cc
myusefulclass.cc
myusefulclass_test.cc

C++ 文件应使用 .cc 文件扩展名，头文件应使用 .h 扩展名.

那些依赖于在特定位置被文本性包含的文件应以 .inc 结尾（另见关于自包含头文件的部分）.

 

不要使用已在 /usr/include 中存在的文件名，例如 db.h.

 

通常情况下，使你的文件名非常具体.

例如，使用 http_server_logs.h 而不是 logs.h .

一个非常常见的情况是一对文件，例如命名为 foo_bar.h 和 foo_bar.cc ，定义一个名为 FooBar 的类.

6.3 类型命名

类型名称以大写字母开头，每个新单词的首字母均大写，不使用下划线： MyExcitingClass ， MyExcitingEnum.

所有类型的名称—— class、struct、using aliases、enum 以及 type template parameters / concept ——遵循相同的命名约定. 类型名应以大写字母开头，并且每个新单词首字母大写。不要使用下划线.

例如：

// classes and structs
class UrlTable { ...
class UrlTableTester { ...
struct UrlTableProperties { ...


// typedefs
typedef hash_map<UrlTableProperties*, std::string> PropertiesMap;


// using aliases
using PropertiesMap = hash_map<UrlTableProperties*, std::string>;


// enums
enum class UrlTableError { ...

注：新项目鼓励不使用 typedefs 而是 using aliases.

6.4 变量命名

变量名（包括函数参数）和数据成员的命名为 snake_case （全部小写，单词间用下划线）.

类的数据成员（但不是 struct 的）还应在末尾加下划线.

例如： a_local_variable ， a_struct_data_member ， a_class_data_member_

 

常用变量名

例如：

xxxxxxxxxx
std::string table_name{};  // OK - snake_case.
xxxxxxxxxx
std::string tableName{};   // Bad - mixed case.

 

类数据成员

类的数据成员（包括静态和非静态）命名方式与普通非成员变量相同，但在名称末尾加下划线.

例外是静态常量类成员，应该遵循常量命名规则.

xxxxxxxxxx
class TableInfo {
 public:
  ...
  static const int kTableVersion {3};  // OK - constant naming.
  ...


 private:
  std::string table_name_{};             // OK - underscore at end.
  static Pool<TableInfo>* pool_{};       // OK.
};

 

结构体数据成员

结构体的成员数据（包括静态和非静态）命名方式与普通非成员变量相同.

它们不使用类中成员数据所带的尾随下划线.

xxxxxxxxxx
struct UrlTableProperties {
  std::string name{};
  int num_entries{};
  static Pool<UrlTableProperties>* pool{};
};

 

常量名

以 constexpr 或 const 声明且其值在程序运行期间保持不变的变量，使用前缀 "k" 并采用混合大小写命名.

在极少数无法通过大小写区分时，可以使用下划线作为分隔符.

例如：

xxxxxxxxxx
const int kDaysInAWeek {7};
const int kAndroid8_0_0 {24};  // Android 8.0.0

所有具有静态存储期的此类变量（即 static 和 global，see Storage Duration）都应以这种方式命名，包括那些作为静态常量类数据成员的变量以及模板中在不同实例化时可能具有不同值的变量.

对于其他存储类的变量（例如自动变量），此约定是可选的；否则应遵循常规的变量命名规则.

例如：

xxxxxxxxxx
void ComputeFoo(absl::string_view suffix) {
  // 这两种都可以
  const absl::string_view kPrefix {"prefix"};
  const absl::string_view prefix  {"prefix"};
  ...
}

6.5 函数命名

通常，函数使用 PascalCase：以大写字母开头，并且每个新单词的首字母都大写.

xxxxxxxxxx
AddTableEntry()
DeleteUrl()
OpenFileOrDie()

对作为 API 一部分并且旨在看起来像函数的类作用域和命名空间作用域常量，适用相同的命名规则，因为它们是对象而非函数这一实现细节并不重要.

访问器和修改器（get 和 set 函数）可以像变量那样命名，如 snake_case.

这些通常对应于实际的成员变量，但这不是必须的.

例如， int count() 和 void set_count(int count).

6.6 命名空间命名

命名空间名称应为 snake_case （全小写，单词之间用下划线分隔）.

在为 namespace 选择名称时，请注意在命名空间外的头文件中使用时名称必须是全限定的，因为未限定的别名（Aliases）通常被禁止.

顶级命名空间必须在全局范围内唯一且易于识别，因此每个命名空间都应由单个项目或团队拥有，名称应基于该项目或团队的名称.

通常，该命名空间下的所有代码应位于一个或多个与命名空间同名的目录下.

嵌套命名空间应避免使用知名顶层命名空间的名称，尤其是 std 和 absl ，因为在 C++ 中，嵌套命名空间并不能防止与其他命名空间中的名称发生冲突.

6.7 枚举命名

枚举量（包括带域的和不带域的枚举）的命名应当像常量，而不是像宏.

也就是说，使用 kEnumName 而不是 ENUM_NAME.

xxxxxxxxxx
enum class UrlTableError {
  kOk,
  kOutOfMemory,
  kMalformedInput,
};

6.8 宏命名

你真的要定义宏吗？

如果要定义，它们应该像这样： MY_MACRO_THAT_SCARES_SMALL_CHILDREN_AND_ADULTS_ALIKE

一般情况下不应使用宏.

然而，如果确实绝对需要使用宏，则应使用全大写字母和下划线命名，并带有项目特定的前缀.
