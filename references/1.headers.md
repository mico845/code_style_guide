## 1. 头文件

通常每个 `.cc` 文件应该有一个配套的 `.h` 文件

常见的例外情况包括单元测试和仅有 `main()` 函数的 `.cc` 文件

---

### 1.1 头文件必须自给自足

> **✅ 规则：**
> 
> - 头文件应该自给自足 (self-contained, 也就是可以**独立编译**), 并以 `.h` 为扩展名. 
> 
> - 给需要被导入 (include) 但不属于头文件的文件设置为 `.inc` 扩展名, 并尽量避免使用.

所有头文件应该自给自足, 

也就是头文件的使用者和重构工具在导入文件时无需任何特殊的前提条件. 

具体来说, 头文件要有头文件防护符并导入它所需的所有其它头文件.



若头文件声明了内联函数 (inline function) 或模版 (template), 

而且头文件的使用者需要实例化 (instantiate) 这些组件时, 

头文件必须直接或通过导入的文件间接提供这些组件的实现 (definition). 

不要把这些实现放到另一个头文件里 (例如 `-inl.h` 文件) 再导入进来; 这是过去的惯例, 但现在被禁止了. 



若模版的所有实例化过程都仅出现在一个 `.cc` 文件中, 比如采用显式 (explicit) 实例化, 或者只有这个 `.cc` 文件会用到模版定义 (definition), 那么可以把模版的定义放在这个文件里.



在少数情况下, 用于导入的文件不能自给自足. 

它们通常是要在特殊的地方导入, 例如另一个文件中间的某处. 

此类文件不需要使用头文件防护符, 也不需要导入它的依赖 (prerequisite). 

此类文件应该使用 `.inc` 扩展名. 尽量少用这种文件, 可行时应该采用自给自足的头文件.

---

### 1.2 #define 守护符

> **✅ 规则：**
>
> - 所有头文件都应该用 `#define` 防护符来防止重复导入.
> - 防护符的格式是: `<项目>_<路径>_<文件名>_H_` .

为保证唯一性，应基于项目源代码树中的完整路径.

例如, `foo` 项目中的文件 `foo/src/bar/baz.h` 应该有如下防护:

```cpp
#ifndef FOO_BAR_BAZ_H_
#define FOO_BAR_BAZ_H_
...
#endif  // FOO_BAR_BAZ_H_
```

---

### 1.3 用什么包含什么

> **✅ 规则：**
>
> - 若代码文件或头文件引用了其他地方定义的符号 (symbol), 该文件应该直接导入 (include) 提供该符号的声明 (declaration) 或者定义 (definition) 的头文件. 
> - 不要依赖间接导入头文件.

不要依赖间接导入. 这样, 人们删除不再需要的 `#include` 语句时, 才不会影响使用者. 

此规则也适用于配套的文件: 若 `foo.cc` 使用了 `bar.h` 的符号, 就需要导入 `bar.h`, 即使 `foo.h` 已经导入了 `bar.h`.

---

### 1.4 前向声明

> **✅ 规则：**
>
> - 尽量避免使用前向声明. 
> - 应该导入你所需的头文件.

---

### 1.5 在头文件中定义函数

> **✅ 规则：**
>
> - 仅当函数定义很短时，才在头文件中于其声明处包含定义.
> - 如果定义有其它理由必须放在头文件中，则将其放在头文件的内部部分（下文解释）.
>
> - 若为了使定义满足 ODR 要求，请用 `inline` 说明符.

只有在函数体很短（比如 10 行或更少）时，才在其 public 声明处定义函数.

较长的函数体应放在 `.cc` 文件中，除非出于性能或技术原因必须放在头文件中.



即使某个定义必须放在头文件中，这也不足以作为把它放在 public 部分的理由（影响阅读公共 API）.

相反，该定义可以放在头文件的内部部分，例如

- 类的 private 部分

- 包含单词 `internal` 的命名空间内

- 放在诸如 `// Implementation details only below here` 的注释之下

  

一旦在头文件中定义，必须通过具有 `inline` 说明符，或通过作为函数模板隐式指定为 inline，或在首次声明时在类体内定义来保证 ODR 安全.

```cpp
template <typename T>
class Foo {
 public:
  int bar() { return bar_; } // short

  void MethodWithHugeBody();

 private:
  int bar_;
};

// Implementation details only below here

template <typename T>
void Foo<T>::MethodWithHugeBody() {
  ...
}
```

### 1.6 `#include` 的路径及顺序

> **✅ 规则：**
>
> - 推荐按照以下顺序导入头文件: 
>
>   1. **配套的头文件**
>
>   2. **C 语言系统库头文件**
>   3. **C++ 标准库头文件**
>   4. **其他库的头文件**
>   5. **本项目的头文件**
>
> - 不能出现 UNIX 目录别名 `.` (当前目录) 或 `..` (上级目录).
>
> - 只有在库要求你这样做时，才应使用尖括号 `<...>` 路径包含头文件

头文件的路径应相对于项目源码目录, 

不能出现 UNIX 目录别名 (alias) `.` (当前目录) 或 `..` (上级目录). 

例如, 应该按如下方式导入 `google-awesome-project/src/base/logging.h`:

```cpp
#include "base/logging.h"
```



只有在库要求你这样做时，才应使用尖括号 `<...>` 路径包含头文件。

特别是，以下头文件需要使用尖括号：

- C 和 C++ 标准库头文件（例如， `<stdlib.h>` 和 `<string>` ）
- POSIX、Linux 和 Windows 系统头文件（例如， `<unistd.h>` 和 `<windows.h>` ）
- 在极少数情况下，third_party 库（例如 `<Python.h>` ）



在 `dir/foo.cc` 或 `dir/foo_test.cc` 这两个实现或测试 `dir2/foo2.h` 内容的文件中, 按如下顺序导入头文件:

> 1. `dir2/foo2.h`.
> 2. 空行
> 3. C 语言系统文件 (确切地说: 用使用方括号和 `.h` 扩展名的头文件), 例如 `<unistd.h>` 和 `<stdlib.h>`.
> 4. 空行
> 5. C++ 标准库头文件 (不含扩展名), 例如 `<algorithm>` 和 `<cstddef>`.
> 6. 空行
> 7. 其他库的 `.h` 文件.
> 8. 空行
> 9. 本项目的 `.h` 文件.

每个非空的分组之间用空行隔开.

这种顺序可以确保在 `dir2/foo2.h` 缺少必要的导入时, 构建 (build) `dir/foo.cc` 或 `dir/foo_test.cc` 会失败. 这样维护这些文件的人会首先发现构建失败, 而不是维护其他库的无辜的人.

`dir/foo.cc` 和 `dir2/foo2.h` 通常位于同一目录下 (如 `base/basictypes_unittest.cc` 和 `base/basictypes.h`), 但有时也放在不同目录下.

注意 C 语言头文件 (如 `stddef.h`) 和对应的 C++ 头文件 (`cstddef`) 是等效的. 新项目鼓励使用 `<c...>` 的 C++ 头文件.

每个分组内部的导入语句应该按字母序排列. 注意旧代码可能没有遵守这条规则, 应该在方便时进行修正.

举例来说, `google-awesome-project/src/foo/internal/fooserver.cc` 的导入语句如下:

> ```cpp
> #include "foo/server/fooserver.h"
> 
> #include <sys/types.h>
> #include <unistd.h>
> 
> #include <string>
> #include <vector>
> 
> #include "base/basictypes.h"
> #include "foo/server/bar.h"
> #include "third_party/absl/flags/flag.h"
> ```

**例外：**

有时，特定系统的代码需要有条件的 includes.

此类代码可以将有条件的 includes 放在其他 includes 之后.

当然，保持特定系统代码的小且局部化。示例：

> ```cpp
> #include "foo/public/fooserver.h"
> 
> #ifdef _WIN32
> #include <windows.h>
> #endif  // _WIN32
> ```
