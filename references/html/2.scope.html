2. 作用域

2.1 命名空间

✅ 规则：

  • 除了少数特殊情况, 应该在命名空间 (namespace) 内放置代码.

  • 命名空间应该有独一无二的名字, 其中包含项目名称, 也可以选择性地包含文件路径.

  • 禁止使用 using-directive (例如 using namespace foo).

  • 禁止使用内联 (inline) 命名空间.

  • 使用单行的嵌套命名空间声明.（例如 namespace my_project::my_component）

建议按如下方法使用命名空间:

  • 用注释终止多行命名空间，如 // namespace mynamespace.

  • 在 include 语句、宏语句和其他命名空间的类的前向声明 (forward declaration) 之后, 用命名空间包裹整个源代码文件:

    ​x
    // In the .h file
    namespace mynamespace {
    

    // 所有声明都在命名空间作用域内
    // 注意在命名空间里没有缩进
    class MyClass {
     public:
      ...
      void Foo();
    };
    

    }  // namespace mynamespace
    xxxxxxxxxx
    // In the .cc file
    namespace mynamespace {
    

    // 函数的定义位于命名空间的作用域内
    void MyClass::Foo() {
      ...
    }
    

    }  // namespace mynamespace

    更复杂的 .cc 文件有更多细节, 比如旗标 (flag) 或 using 声明.

    xxxxxxxxxx
    #include "a.h"
    

    ABSL_FLAG(bool, someflag, false, "a flag");
    

    namespace mynamespace {
    

    using ::foo::Bar;
    

    ...code for mynamespace...    // 代码贴着左边缘
    

    }  // namespace mynamespace

  • 不要在 std 命名空间内声明任何东西.

    不要前向声明 (forward declare) 标准库的类.

    在 std 命名空间内声明实体是未定义行为 (undefined behavior), 也就是会损害可移植性.

    若要声明标准库的实体, 应该导入对应的头文件.

  • 禁止使用 using-directive 引入命名空间的所有符号。

    xxxxxxxxxx
    // 禁止: 这会污染命名空间.
    using namespace foo;

  • 除了在明显标注为内部使用的命名空间内, 不要让头文件引入命名空间别名 (namespace alias).

    这是因为头文件的命名空间中引入的任何东西都是该文件的公开 API.

    xxxxxxxxxx
    // 在 .h 文件中，别名不得作为独立的 API，或者必须隐藏在实现细节中(如 namespace internal)
    namespace librarian {
    

    namespace internal {  // Internal, not part of the API.
    namespace sidetable = ::pipeline_diagnostics::sidetable;
    }  // namespace internal
    

    inline void my_inline_function() {
      // Local to a function.
      namespace baz = ::foo::bar::baz;
      ...
    }
    

    }  // namespace librarian
    xxxxxxxxxx
    // 删除 .cc 文件中某些常用名称的无关部分
    namespace sidetable = ::pipeline_diagnostics::sidetable;

  • 禁止内联命名空间.

  • 如果命名空间的名称包含 “internal”, 代表用户不应该使用这些 API.

    xxxxxxxxxx
    // Absl 以外的代码不应该使用这一内部符号.
    using ::absl::container_internal::ImplementationDetail;

    请注意，嵌套 internal namespace 中的库之间仍存在命名冲突的风险，因此应通过添加库的文件名为命名空间内的每个库赋予唯一的 internal namespace.

    例如， gshoe/widget.h 应使用 gshoe::internal_widget 而不是仅使用 gshoe::internal.

  • 我们鼓励新的代码使用单行的嵌套命名空间声明.

    xxxxxxxxxx
    namespace my_project::my_component {
    

      ...
    

    }  // namespace my_project::my_component

2.2 内部链接

✅ 规则：

  • 若其他文件不需要使用 .cc 文件中的定义, 这些定义可以放入匿名命名空间 (unnamed namespace) 或声明为 static, 以实现内部链接 (internal linkage). 但是不要在 .h 文件中使用这些手段.

  • 新项目鼓励使用匿名命名空间而非 static.

建议 .cc 文件中所有不需要外部使用的代码采用内部链接. 不要在 .h 文件中使用内部链接.

 

匿名命名空间的声明应与具名命名空间的格式相同.

在末尾的注释中, 不用填写命名空间名称:

xxxxxxxxxx
namespace {
...
}  // namespace

2.3 非成员函数、静态成员函数和全局函数

✅ 规则：

  • 建议将非成员 (nonmember) 函数放入命名空间.

  • 尽量不要使用完全全局的函数 (completely global function).

  • 不要仅仅为了给静态成员 (static member) 分组而使用类 (class). 类的静态方法应当和类的实例或静态数据紧密相关.

2.4 局部变量

✅ 规则：

  • 应该尽可能缩小函数变量的作用域 (scope), 例如：

    • 声明离第一次使用的位置越近越好.

    • 应该在语句内声明用于 if、switch、while 和 for 语句的变量

  • 在声明的同时初始化.

你可以在 C++ 函数的任何位置声明变量.

我们提倡尽可能缩小变量的作用域, 且声明离第一次使用的位置越近越好.

这样读者更容易找到声明, 了解变量的类型和初始值.

特别地, 应该直接初始化变量而非先声明再赋值, 比如:

xxxxxxxxxx
int i;
i = f();     // 不好: 初始化和声明分离
xxxxxxxxxx
int i {f()}; // 良好: 声明时初始化
xxxxxxxxxx
int jobs {NumJobs()};
// 更多代码...
f(jobs);      // 不好: 初始化和使用位置分离
xxxxxxxxxx
int jobs {NumJobs()};
f(jobs);      // 良好: 初始化以后立即 (或很快) 使用
xxxxxxxxxx
vector<int> v;
v.push_back(1);  
v.push_back(2);
xxxxxxxxxx
vector<int> v {1, 2}; // 良好: 立即初始化 v

 

通常应该在语句内声明用于 if、switch、while 和 for 语句的变量, 这样会把作用域限制在语句内.

例如:

xxxxxxxxxx
while (const char* p = strchr(str, '/')) str = p + 1;
xxxxxxxxxx
if (auto it = mp.find("key"); it != mp.end())
        return it->second;
xxxxxxxxxx
switch (auto dev = Device{}; dev.state())
    {
        case Device::SLEEP:
            /*...*/
            break;
        case Device::READY:
            /*...*/
            break;
        case Device::BAD:
            /*...*/
            break;
    }

需要注意的是, 如果变量是一个对象, 那么它每次进入作用域时会调用构造函数, 每次退出作用域时都会调用析构函数.

xxxxxxxxxx
// 低效的实现:
for (int i = 0; i < 1000000; ++i) {
    Foo f;  // 调用 1000000 次构造函数和析构函数.
    f.DoSomething(i);
}
xxxxxxxxxx
Foo f;  // 调用 1 次构造函数和析构函数.
for (int i = 0; i < 1000000; ++i) {
    f.DoSomething(i);
}

2.5 静态和全局变量

✅ 规则：

  • 对于静态存储期的变量：

    1. 能 constexpr 就使用 constexpr

    2. 需要可变状态时，优先使用 constinit

    3. 禁止非局部静态变量发生动态初始化，除非经过明确审查

    4. 默认禁止非平凡析构的静态存储期对象

    5. 局部静态变量（函数内 static 变量）可接受动态初始化，但仍应审查析构行为

对于静态存储期的变量（全局 / namespace-scope / static 成员变量 / 函数内 static 变量）：

  1. 常量 -> 优先 constexpr

    xxxxxxxxxx
    constexpr int kMaxSize{ 1024 };
    constexpr std::array<int, 3> kValues{ 1, 2, 3 };
    constexpr std::string_view kName{ "isp" };

    constexpr 变量必须常量初始化，并且可用于常量表达式.

  2. 非常量(能不用就不用) -> 优先 constinit

    xxxxxxxxxx
    constinit int g_counter {0};

    constinit -> 强制静态初始化

    constinit 只约束初始化方式，不表示只读，也不表示可用于常量表达式.

  3. 非平凡析构的静态存储期的变量：默认禁止

    平凡析构是指析构函数 (destructor) 不会做任何事情, 包括成员和基类 (base) 的析构函数.正式地说, 就是这一类型 (type) 没有用户提供的析构函数或虚析构函数 (virtual destructor), 且所有成员和基类也能平凡地析构. 

    xxxxxxxxxx
    const int kNum {10};  // 平凡析构
    

    struct X { int n; };
    const X kX[] {{1}, {2}, {3}};  // 平凡析构
    

    void foo() {
      static const char* const kMessages[] {"hello", "world"};  // 平凡析构
    }
    xxxxxxxxxx
    // 非平凡的析构.
    const std::string kFoo {"foo"};
    

    // 和上面相同的原因, 即使 kBar 是引用 (该规则也适用于生命周期被延长的临时对象).
    const std::string& kBar {StrCat("a", "b", "c")};
    

    void bar() {
      //  非平凡的析构.
      static std::map<int, int> kData {{1, 0}, {2, 0}, {3, 0}};
    }

    在 C++23 中，constexpr 不严格等于平凡析构.

    因此对于类类型的静态存储期对象，

    仍应单独审查其析构行为(可以使用 std::is_trivially_destructible_v<T>).

    例如：

    xxxxxxxxxx
    static_assert(std::is_trivially_destructible_v<decltype(kValues)>);

 

常用的语法结构

  • 全局字符串

    如果你需要具名的 (named) 全局或静态字符串常量, 可以采用 constexpr 修饰的 string_view 变量、字符数组或指向字符串字面量 (literal) 的字符指针.字符串字面量具有静态储存周期, 因此通常能满足需要.

  • 字典（Map）和集合（Set）等动态容器

    若你需要用静态变量储存不会改变的数据 (例如用于搜索的集合或查找表), 不要使用标准库的动态容器, 因为这些容器拥有非平凡的析构函数.

    可以考虑用平凡类型的数组替代, 例如 int 数组的数组 (作为把 int 映射到 int 的字典) 或者数对 (pair) 的数组 (例如一组 int 和 const char* 的数对).

    对于少量数据, 线性搜索就足够了, 而且因为具有内存局部性 (memory locality) 而更加高效;

    可以使用 absl/algorithm/container.h 中的工具实现常见操作.

    如有需要, 可以保持数据有序并采用二分查找法 (binary search).

    如果你确实需要使用标准库的动态容器, 建议使用如下文所述的函数内局部静态指针.

  • 智能指针 (smart pointer, 例如 std::unique_ptr 和 std::shared_ptr)

    在析构时有释放资源（delete）的操作, 因此它本身是非平凡析构的，不能作为静态变量.

    请思考你的情景是否适用于本小节描述的其他模式. 简单的解决方式是, 用裸指针 (plain pointer) 指向动态分配的对象, 并且永远不删除这个对象 (参见最后一点).

  • 自定义类型的静态变量

    如果静态数据或常量数据是自定义类型, 请给这一类型设置平凡的析构函数和 constexpr 修饰的构造函数.

  • 若以上都不适用, 你可以采用函数内局部静态指针或引用, 动态分配一个对象且永不删除 (例如 static const auto& impl = *new T(args...);).
